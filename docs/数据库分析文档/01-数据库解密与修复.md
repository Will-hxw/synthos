# 01 - 数据库解密与修复

> **前置知识**：QQ NT 使用 **SQLCipher** 加密 SQLite 数据库。解密需要先获取数据库密钥，然后通过 SQLCipher 工具解密。

---

## 1. 基本原理

### 1.1 加密方案

QQ NT 数据库基于 SQLCipher 加密，关键参数：

| 参数 | 值 | 说明 |
|:---|:---|:---|
| `cipher` | `aes-256-cbc` | 加密算法 |
| `kdf_iter` | `4000` | PBKDF2 迭代次数（SQLCipher 默认为 256000，QQ 改为 4000） |
| `cipher_page_size` | `4096` | 页大小 |
| `cipher_hmac_algorithm` | `HMAC_SHA1` | HMAC 算法 |
| `cipher_default_kdf_algorithm` | `PBKDF2_HMAC_SHA512` | KDF 算法 |

### 1.2 文件头冗余数据

**每个 `.db` 文件头部有 1024 字节的无用内容**，解密前需先去除：

```bash
# Linux / macOS
cat nt_msg.db | tail -c +1025 > nt_msg.clean.db

# Windows PowerShell
$content = [System.IO.File]::ReadAllBytes("nt_msg.db")
[System.IO.File]::WriteAllBytes("nt_msg.clean.db", $content[1024..($content.Length - 1)])

# 或使用 Python
python -c "open('nt_msg.clean.db','wb').write(open('nt_msg.db','rb').read()[1024:])"
```

## 2. 各平台密钥获取

### 2.1 Windows

函数 `nt_sqlite3_key_v2` 负责调用 SQLCipher 的 `sqlite3_key_v2`，密钥以明文形式传入。

**方法一：Frida Hook**

Windows 下特征 Hex（QQ 9.9.1.15043）：
```
48 89 5C 24 08 48 89 6C 24 10 48 89 74 24 18 57
48 83 EC 20 41 8B F9 49 8B F0 4C 8B CA 4C 8B C1
48 8B EA 48 8B D9 48 8D 15 33 05 A0 00 B9 08 00
```

`sqlite3_key_v2` 函数签名：
```c
int sqlite3_key_v2(
  sqlite3 *db,          // 数据库句柄
  const char *zDbName,  // 数据库名称
  const void *pKey,     // 密钥（明文）
  int nKey              // 密钥长度
);
```

Hook 第三个参数 `pKey` 即可获取明文密钥。

**方法二：PowerShell 一键脚本**

参见 [windows_ntqq_get_key.ps1](https://github.com/QQBackup/qq-win-db-key/blob/master/windows_ntqq_get_key.ps1)。

**方法三：nt-hook**

参见 [msojocs/nt-hook](https://github.com/msojocs/nt-hook)（需 Node.js 环境编译）。

### 2.2 macOS

**Apple Silicon (ARM64)**：

1. 使用 Hopper Disassembler 打开 `wrapper.node`
2. 选择 `aarch64` 架构
3. 搜索 `nt_sqlite3_key_v2`
4. 定位到调用 `sqlite3_key_v2` 的函数
5. 通过 `x2` 寄存器获取密钥地址（ARM64 调用约定中 `pKey` 参数位于 `x2`）

**Intel (x86_64)**：

类似流程，但寄存器为 `rdx`（System V AMD64 ABI）。

### 2.3 Android

Android 端密钥由 `QQ_path_hash` 计算得出，流程较复杂。参考 [QQDecrypt Android 教程](https://docs.aaqwq.top/) 获取 `nt_uid` 和 `rand` 后计算。

### 2.4 Linux / iOS

基本流程与对应 CPU 架构的 macOS/Android 方案一致，参考对应平台的 Frida 或反编译教程。

## 3. 数据库解密操作

### 3.1 命令行方式（sqlcipher CLI）

```bash
# 1. 打开已去除文件头的数据库
sqlcipher nt_msg.clean.db

# 2. 配置解密参数并输入密钥
sqlite> PRAGMA key = 'your_pass_key_here';
sqlite> PRAGMA kdf_iter = 4000;

# 3. 验证数据库完整性
sqlite> .tables

# 4. 导出为无加密数据库
sqlite> ATTACH DATABASE 'nt_msg.decrypted.db' AS plaintext KEY '';
sqlite> SELECT sqlcipher_export('plaintext');
sqlite> DETACH DATABASE plaintext;
sqlite> .exit
```

### 3.2 SQLiteStudio 方式（GUI）

1. 用 SQLiteStudio 打开 `nt_msg.clean.db`
2. 在"数据库配置"中输入：

```sql
PRAGMA key = 'pass_key';
PRAGMA cipher_page_size = 4096;
PRAGMA kdf_iter = 4000;
PRAGMA cipher_hmac_algorithm = HMAC_SHA1;
PRAGMA cipher_default_kdf_algorithm = PBKDF2_HMAC_SHA512;
PRAGMA cipher = 'aes-256-cbc';
```

3. 导出为 SQL 文件，再通过 sqlite3 CLI 重新生成：

```bash
sqlite3 nt_msg.decrypted.db ".read nt_msg.sql"
```

### 3.3 Python 方式

```python
# 使用 pysqlcipher3 或 sqlcipher3
from pysqlcipher3 import dbapi2 as sqlcipher

# 去除文件头 1024 字节
with open('nt_msg.db', 'rb') as f:
    data = f.read()[1024:]
with open('nt_msg.clean.db', 'wb') as f:
    f.write(data)

# 解密
conn = sqlcipher.connect('nt_msg.clean.db')
conn.execute("PRAGMA key = 'pass_key'")
conn.execute("PRAGMA kdf_iter = 4000")
conn.execute("ATTACH DATABASE 'nt_msg.decrypted.db' AS plaintext KEY ''")
conn.execute("SELECT sqlcipher_export('plaintext')")
conn.execute("DETACH DATABASE plaintext")
conn.close()
```

## 4. 数据库修复

如果解密时出现 `Runtime error: database disk image is malformed` 错误：

### 4.1 检查完整性

```sql
PRAGMA integrity_check;
```

### 4.2 修复流程

```bash
# 1. 导出所有 SQL 语句
sqlcipher nt_msg.clean.db
sqlite> PRAGMA key = 'pass_key';
sqlite> PRAGMA kdf_iter = 4000;
sqlite> .output dump_all.sql
sqlite> .dump
sqlite> .exit

# 2. 去除事务语句（ROLLBACK 等）
cat dump_all.sql | grep -v TRANSACTION | grep -v ROLLBACK | grep -v COMMIT > dump_all_notrans.sql

# 3. 重新生成数据库
sqlite3 repair.db ".read dump_all_notrans.sql"
```

注意：部分数据可能因损坏而丢失，修复后需验证关键表的数据完整性。

---

## 5. 注意事项

1. **密钥安全**：数据库密钥是敏感信息，请勿泄露。
2. **版本差异**：不同 QQ 版本的特征 Hex 可能不同，需在目标版本上重新定位。
3. **多数据库密钥**：不同 `.db` 文件可能使用不同密钥，必须逐个解密。
4. **平台差异**：Android 端的 `gpro_v1-6_{nt_uid}.db` 暂未实现解密。
5. **法律合规**：仅供学习交流使用，严禁用于违反法律法规的行为。
